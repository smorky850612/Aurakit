# AuraKit — Discovery Protocol (Step-back Before Build)

> BUILD/FIX 모드 실행 전 필수 실행. 요구사항을 명확히 하고 불필요한 작업을 사전 차단.
> Superpowers Framework의 "Step-back Discovery" 방법론 기반.

---

## 원칙

- **코딩 전 이해**: 무엇을, 왜, 어떻게 만들지 먼저 확인
- **YAGNI 검증**: 실제로 필요한가? 더 단순한 해결책은 없나?
- **모호성 제거**: 불명확한 요구사항은 구현 전에 해결
- **Mandatory**: 이 단계는 생략 불가. 명확한 답변이 나올 때까지 진행

---

## 5-Question Discovery Framework

사용자 요청을 받으면 구현 전에 다음 5가지를 내부적으로 검토하고, 모호한 항목이 있을 때만 사용자에게 확인:

### Q1. 실제 문제는 무엇인가?

```
요청: "로그인 기능 만들어줘"
→ 표면: 로그인 UI + API 구현
→ 실제: 사용자 인증 + 세션 관리 + 보호 라우트
→ 확인: "이메일/비밀번호 로그인만 필요한가요, 소셜 로그인(Google/GitHub)도 필요한가요?"
```

### Q2. 이미 있는 것은 없는가?

```
→ 기존 auth 라이브러리/미들웨어 확인
→ 유사 기능 중복 구현 방지
→ Scout 에이전트로 프로젝트 탐색 후 판단
```

### Q3. 최소 구현으로 목표를 달성할 수 있는가?

```
YAGNI 체크리스트:
  □ 지금 당장 필요한가? (vs 나중에 필요할 것 같은가)
  □ 더 단순한 대안이 있는가?
  □ 외부 라이브러리로 대체 가능한가?
  □ 기존 코드 수정으로 해결 가능한가?
```

### Q4. 구현 범위는 어디까지인가?

```
포함:
  - 명시적으로 요청된 기능
  - 동작에 필수적인 의존성

제외 (별도 작업으로 분리):
  - 요청하지 않은 추가 기능
  - "나중에 필요할 것 같은" 기능
  - 현재 버그와 무관한 리팩토링
```

### Q5. 성공 기준은 무엇인가?

```
명확한 완료 조건:
  - 어떤 동작이 되면 완성인가?
  - 어떤 경우에 실패여야 하는가?
  - 테스트할 시나리오는?
```

## Discovery 출력 포맷

분석 완료 후 구현 시작 전에 출력:

```
🔍 Discovery — [요청 요약]

이해한 요구사항:
  ✓ [핵심 기능 1]
  ✓ [핵심 기능 2]

제외 범위:
  ✗ [요청되지 않은 것 1] — (이유: 범위 외)
  ✗ [요청되지 않은 것 2] — (이유: 별도 작업)

성공 기준:
  → [완료 조건 1]
  → [완료 조건 2]

최소 구현: [N]개 파일
대안 검토: [더 단순한 방법이 있으면 언급, 없으면 생략]
```

## 확인 질문 기준

사용자에게 질문하는 경우 (모호성이 명확히 있을 때만):

```
질문하는 경우 ✅:
  - 기술 스택이 불명확 (Next.js? Express? FastAPI?)
  - 인증 방식이 불명확 (이메일? 소셜? 모두?)
  - 데이터 구조가 불명확하여 DB 스키마를 결정할 수 없음
  - 2가지 이상의 완전히 다른 구현 방향이 존재

질문하지 않는 경우 ❌:
  - 일반적인 구현 방식이 명확히 존재
  - 프로젝트 프로필에서 스택이 이미 파악됨
  - 사소한 선택 (변수명, 파일 구조 등)
  - 사용자가 빠른 결과를 원하는 경우
```
