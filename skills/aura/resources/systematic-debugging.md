# AuraKit — Systematic Debugging (4-Phase)

> `/aura debug:` 호출 시 로딩. 5-WHY 기반 근본 원인 분석.

---

## 4-Phase 디버깅 프로세스

### Phase 1 — Observe (현상 파악)

```bash
# 에러 메시지 전체 캡처
# 스택 트레이스 확인
# 재현 조건 파악 (항상/가끔/특정 조건)
```

출력:
```
🔴 DEBUG — [기능/컴포넌트명]

에러: [에러 메시지]
위치: [파일:라인]
재현: [항상 / 조건 X일 때 / 가끔]
최초 발생: [최근 변경 사항]
```

---

### Phase 2 — Hypothesize (5-WHY 분석)

근본 원인이 나올 때까지 "왜?"를 5번 반복:

```
왜-1: 왜 에러가 발생했는가?
  → [직접 원인]

왜-2: 왜 [직접 원인]이 발생했는가?
  → [중간 원인]

왜-3: 왜 [중간 원인]이 발생했는가?
  → [더 깊은 원인]

왜-4: 왜 [더 깊은 원인]이 발생했는가?
  → [구조적 원인]

왜-5: 왜 [구조적 원인]이 존재하는가?
  → 근본 원인: [최종 진단]
```

---

### Phase 3 — Fix (최소 변경 수정)

원칙:
- **최소 변경**: 근본 원인만 수정, 주변 코드 변경 금지
- **단일 수정**: 여러 문제를 동시에 수정하지 않음
- **검증 가능**: 수정 후 재현 테스트 가능한 형태로

```
🔧 수정 계획:
파일: [파일경로]
변경: [변경 내용 1줄]
영향 범위: [영향받는 다른 파일/기능]
```

---

### Phase 4 — Verify (검증)

```bash
# 에러 재현 시도 (수정 전과 동일 조건)
# 관련 엣지 케이스 확인
# 회귀 테스트 (기존 기능 영향 없음 확인)
```

출력:
```
✅ DEBUG 완료

근본 원인: [5-WHY 최종 진단]
수정 내용: [변경 파일 + 내용]
재발 방지: [구조적 개선 제안]
```

---

## 디버깅 패턴별 접근

| 패턴 | 접근 |
|------|------|
| TypeError/ReferenceError | 타입 체크, null 체크, 모듈 임포트 순서 |
| 비동기 에러 | await 누락, Promise 체인, race condition |
| 빌드 에러 | 의존성 버전, 타입 정의, 설정 파일 |
| 런타임 에러 | 환경 변수, 권한, 네트워크 연결 |
| 간헐적 에러 | 타이밍 문제, 캐시, 상태 경쟁 |

---

## 에스컬레이션 기준

3번 이상 5-WHY 분석 후 근본 원인 미확인 → `resources/review-pipeline.md`의 SecurityAgent + Reviewer 병렬 투입.
